# robot_vs

本仓库用于多机器人对抗场景（红蓝对抗）的仿真与现实部署实验。系统中包含多个小车（机器人）agent，在红方和蓝方阵营下分别由各自的 **manager 节点** 进行统筹与决策，小车本体只负责执行基础动作和指令解析。

## 功能概述

- 支持 **仿真环境** 与 **现实环境** 下的多机器人独立运行  
- 采用 **红方 / 蓝方 两个阵营** 的对抗结构
  - 每个阵营有一个 manager 负责：
    - 管理本阵营所有小车
    - 决策、感知与任务分配
  - 每辆小车只负责：
    - 基本运动控制接口（如速度控制、定位消息、导航功能等）
    - 上位机 / manager 下发指令的解析与执行

## 多机方面技术要点

### 1. 多机器人命名空间与话题管理

- 为每个小车的 ROS 节点与话题加上 **命名空间 (namespace)**，实现多机器人之间的话题隔离  
  - 例如：`/red1/cmd_vel`、`/blue2/odom` 等
- 对部分话题使用 **remap**，在不修改底层代码的前提，灵活适配不同机器人实例与仿真/现实环境
- 通过命名空间与 remap 的组合，实现：
  - 多个相同功能的小车并行运行
  - 防止话题名冲突
  - 简化仿真与现实配置切换

### 2. TF 坐标系前缀隔离

- 为所有 TF 变换增加 **前缀 (prefix)**，使得每辆小车的 TF 树互相独立  
  - 例如：`red1/base_link`、`blue2/base_link`
- 利用前缀隔离不同机器人之间的坐标系，避免 TF 冲突和混淆
- 在 manager 层面可以根据前缀选择性地订阅和使用对应小车的 TF 信息

## 仿真与现实部署

### 仿真环境

- 在gazebo仿真中，多个小车 agent 通过命名空间和 TF 前缀实现完全独立运行
- 红方 / 蓝方 manager 可以在仿真中实现策略开发与对抗算法验证
- 仿真与现实在话题结构上尽量保持一致，便于算法迁移
- 有编辑好的rviz界面

### 现实环境（进行中）

- 现实系统正在制作与调试中，设计目标为：
  - 所有机器人与上位机共用 **同一个 rosmaster**
  - 主机（上位机 / 管理机）与从机（车载计算单元）在 **同一局域网** 内通信
  - 继续沿用仿真中的 **命名空间 + TF 前缀** 设计，保证多车并行运行与话题隔离
